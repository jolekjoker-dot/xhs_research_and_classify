# Agent 面试常见问题与参考答案

> 整理自小红书面经（字节跳动、快手等大厂真实面试题），仅供复习参考。

## 一、Agent 核心概念与架构

### Q1：Agent 的基本架构组成是什么？与 LLM Chain 的区别？

**答：** Agent = LLM + 规划(Planning) + 记忆(Memory) + 工具使用(Tool Use)。

- **Chain** 是预定义的、线性的硬编码工作流，步骤固定。
- **Agent** 具备"自主性"，根据目标自发决定执行路径，通过推理循环（Reasoning Loop）不断调整策略。

### Q2：ReAct 模式的工作原理？

**答：** ReAct = Reasoning + Acting，是 Agent 的基石。

流程：LLM 先生成一段推理（Thought）→ 说明下一步要做什么 → 调用工具执行（Action）→ 观察结果（Observation）→ 根据结果进入下一轮推理，如此循环直到任务完成。

### Q3：ReAct vs Plan-and-Execute 各有什么优劣？如何选择？

**答：**

| | ReAct | Plan-and-Execute |
|---|---|---|
| 优点 | 灵活度高，用户中途改需求也能跟上 | 省 Token，适合确定性任务 |
| 缺点 | Token 消耗大 | 中间出岔子不好处理 |

**实战做法：** 混合使用。大体上先有个计划，执行细节里遇到异常再切到 ReAct 模式局部调整。

### Q4：Workflow 与 Autonomous Agent 的区别？2026 年趋势？

**答：**

- **Workflow：** 通过 DAG 或状态机硬编码路径，高可靠性、结果可预期，适用于报销审批、标准化客服。
- **Autonomous Agent：** 由 LLM 决定循环次数和工具调用，灵活性极高，适用于开放式研究、代码编写。
- **2026 工程趋势：** "用 Workflow 约束 Agent"——在框架定义的路径内给予 Agent 局部决策权。

### Q5：Orchestrator-Workers 模式是什么？

**答：** 主 Agent（Orchestrator）负责将复杂任务分解为子任务，分发给具有不同 Skill 的 Worker Agents，最后汇总结果。

- **适用场景：** 大型软件开发（一个写 UI，一个写后端，一个写测试）。
- **难点：** 任务分解粒度——太细通信成本高，太粗 Worker 产生幻觉。

### Q6：Reflection/Self-Correction 模式是什么？

**答：** 这是提升 Agent 成功率最有效的模式。Agent 生成输出后，由另一个（或同一个）Agent 扮演 Critic，检查输出是否符合约束条件，并提供反馈让前者迭代。

**技术细节：** 可使用 Reflexion 架构，记录"失败轨迹"作为长短期记忆，避免重复犯同样的错误。

### Q7：Multi-Agent 选型依据是什么？

**答：** 这是面试高频题，需要结合实际项目场景回答：

- 任务复杂度：单一 Agent 能否胜任？是否需要不同专业能力？
- 任务并行度：能否拆分为独立子任务并行执行？
- 质量要求：是否需要多角色校验（如 coder + reviewer）？
- 系统的可维护性和可观测性

## 二、提示词与上下文工程

### Q8：提示词模板是怎么构建的？

**答：** （根据项目经验回答，关注以下要点）

- 模板的变量化设计：哪些部分固定、哪些动态注入
- 模板的版本管理和 A/B 测试
- 不同场景（角色扮演/工具调用/推理）对应不同模板结构
- 如何验证模板效果

### Q9：上下文工程是怎么设计的？

**答：** 核心关注点：

- **上下文拼接结构：** system prompt + 历史对话 + 检索到的知识 + 当前 query
- **防过长的策略：** 滑动窗口截断历史、对长文本做摘要压缩、按相关性排序后截断
- **To-Do List 的作用：** 让模型把复杂任务拆成步骤并显式记录，每一步完成后打勾。这样模型能明确知道"当前进展到哪了"，减少遗漏和幻觉。

### Q10：Skills 的原理有没有了解过？

**答：** Skills 本质上是动态注入的领域知识/指令片段：

- 按需加载：根据用户意图匹配对应 Skill，将其内容注入上下文
- 结构化的提示词模板，包含角色定义、操作步骤、输出格式
- 好处：避免一次性加载所有指令撑爆上下文窗口

## 三、记忆机制

### Q11：Agent 的记忆怎么实现？短期 vs 长期？

**答：**

**短期记忆：**
- 当前会话的消息记录（Chat History），直接利用 Context Window
- 状态变量：当前步骤、中间结果等，可存 Redis
- 控制长度避免撑爆窗口

**长期记忆：**
- 通过 RAG 实现：将历史经验、知识编码为 Embedding 存入向量数据库
- Agent 执行任务前检索相关经验（Experience Retrieval）
- 会话结束后压缩成摘要，或抽取用户偏好/常用信息存入向量库

**2026 新趋势：**
- 利用长文本模型（Long-context LLMs）直接处理超长历史
- "摘要层级结构"对记忆进行递归压缩

### Q12：记忆更新策略是什么？

**答：**

- 增量更新：新信息追加存储
- 合并去重：相似记忆合并，避免冗余
- 衰减机制：根据时间和使用频率降低旧记忆权重
- 触发式更新：关键事件触发记忆刷新

## 四、RAG 与检索

### Q13：为什么引入父子索引？

**答：** （需要结合实际场景）

- 父索引做粗筛：快速定位相关文档范围
- 子索引做精排：在文档内部精确匹配段落/句子
- 好处：兼顾检索效率和精确度

### Q14：为什么引入 BM25？和向量检索如何组合？

**答：**

- **BM25 优势：** 对关键词匹配敏感，处理专有名词、缩写、精确术语等场景优于向量检索
- **向量检索优势：** 语义理解强，处理同义词、改写表达
- **组合方式：** 混合检索，BM25 结果和向量检索结果按一定比例融合（如 3:7 或根据场景动态调整）

**整体检索流程：** Query 输入 → 并行执行 BM25 + 向量检索 → 结果融合（RRF 或加权融合）→ Rerank → TopK 截断 → 拼接上下文

### Q15：Rerank 相关策略？

**答：**

- Rerank 后一般返回 5-10 个块，具体数量根据模型上下文窗口和相关性分数阈值确定
- TopK 截断：通常取 5-8 条，不超过上下文窗口的 60-70%
- 过长则截断或压缩，过短则降低截断阈值或回退到原始检索结果补充
- 使用 Cross-Encoder 模型（如 bge-reranker）对初检结果重排序

### Q16：RAG 如何评测？

**答：**

**评测维度：**
- 检索质量：Recall@k、Precision@k、MRR、NDCG
- 生成质量：Faithfulness（忠实度）、Answer Relevance、Context Relevance
- 端到端：ROUGE、BLEU（参考）、人工评测

**评测数据集构建：**
- 覆盖常见用户问题类型
- 包含边界/对抗性样本
- 标注标准答案和关键信息点
- 持续从线上 badcase 补充

### Q17：如何提升 RAG 相关度和回答效果？

**答：**

- **检索阶段：** 优化索引粒度、引入混合检索、Query 改写/扩展、HyDE
- **生成阶段：** 优化 Prompt 模板、引入 Citation 机制、多轮验证
- **验证方式：** A/B 测试对比 + 离线评测指标 + 人工抽样

## 五、Function Calling 与工具调用

### Q18：Function Calling 怎么设计？

**答：**

- 工具定义：名称、描述、参数 Schema（类型、必填项、约束）
- 返回格式：结构化 JSON，包含执行结果和状态
- 错误处理：统一封装，返回结构化错误信息
- 调用编排：支持并行/串行调用，处理依赖关系

### Q19：怎么让模型不瞎编参数？

**答：**

1. 用模型自带的 Function Calling，返回结构化数据，直接解析（最稳）
2. 工具定义写清楚：参数类型、必填项、示例都给足
3. Prompt 中硬性要求输出 JSON，加正则解析和格式校验，不对就重试（最多两次）
4. 关键参数在后端设默认值兜底，不能全指望模型

### Q20：工具调用失败/超时怎么办？

**答：**

1. 所有工具调用封装成统一函数
2. 捕获异常后返回结构化错误信息（如 "Error: 超时"）
3. 把错误喂回模型，让它自己决定：重试 / 换工具 / 告知用户
4. 重试次数限制两次，整体超时设上限（如 30 秒）
5. 关键工具准备备用 API，主挂了就降级

### Q21：多工具调用时如何决定调用顺序？

**答：**

- 由模型推理决定（Autonomous）：模型分析依赖关系后自主安排
- 规则前置（Workflow）：预设工具调用链，适用于确定性强场景
- 混合模式：框架层面定义依赖 DAG，单步内由模型自主决策

### Q22：工具调用的安全控制怎么实现？

**答：**

- 权限控制：不同工具分级授权
- 白名单机制：限制可调用的工具和接口范围
- 输入校验：工具调用前验证参数合法性和边界值
- 敏感操作拦截：写操作、删除操作需额外确认
- 调用频率限制：防止滥用和恶意循环

## 六、任务规划与多智能体协作

### Q23：Agent 的任务规划怎么做？

**答：**

- **模型驱动规划：** 让 LLM 自行分析任务并生成步骤（ReAct 循环）
- **规则驱动规划：** 预设步骤流程，模型只负责执行
- **混合模式：** 框架层面制定大步骤（规则），步骤内由模型自主规划细节

### Q24：多智能体怎么协作？

**答：**

1. **角色定义：** 每个 Agent 定死角色（如程序员只写代码，审查员只挑毛病），System Prompt 限定职责和输出格式
2. **协作方式：** 顺序链，前一个输出直接作为下一个的输入
3. **消息格式：** 用 JSON 串起来，带上任务 ID 方便追踪
4. **冲突处理：** 加仲裁者 Agent 或关键步骤人工介入

### Q25：Multi-Agent 如何评测？

**答：**

- 单 Agent 评测：任务完成率、响应质量、工具调用准确率
- Multi-Agent 评测：协作效率、任务分解合理性、冲突处理能力、端到端成功率
- 需要构建多轮交互的评测数据集，模拟真实协作场景

## 七、意图识别与查询改写

### Q26：意图识别怎么做？如何评测？

**答：**

- **实现：** 分类模型（预定义意图类别）或 LLM few-shot 推理
- **并行化：** 多个意图分类器并行运行，取置信度最高的结果，减少延迟
- **评测指标：** 准确率、召回率、F1、混淆矩阵分析
- **数据标注：** 从线上日志采样 + 人工标注 + 持续补充误判 case

### Q27：多维度的查询改写是什么？

**答：**

- 拼写纠错、同义词替换、指代消解
- 复杂查询拆解为多个子查询
- 需要用户补充信息时：生成追问 → 等待用户回复 → 补全改写
- 技术上通过 LLM + 规则引擎组合实现

## 八、Prompt 注入与安全防护

### Q28：Prompt 注入攻击如何防御？

**答：**

1. **输入过滤：** 对用户输入做正则匹配和语义检测，识别注入模式
2. **规则校验：** 检测分隔符注入、角色覆盖、指令劫持等常见攻击
3. **权限隔离：** 用户输入和 System Prompt 在位置上严格分离，System Prompt 放最后
4. **输出审查：** 对模型输出做合规检查
5. **持续对抗测试：** 定期用新的注入手法测试防线的有效性

## 九、限流算法

### Q29：分布式令牌桶限流原理？

**答：**

- 令牌以固定速率生成，放入桶中
- 桶有容量上限，满则丢弃多余令牌
- 请求到达时消耗令牌，无令牌则拒绝
- 分布式场景下使用 Redis 存储令牌计数 + Lua 脚本保证原子性

### Q30：漏桶算法 vs 滑动窗口 vs 令牌桶？

**答：**

- **漏桶：** 固定速率处理，超出则丢弃；平滑流量但无法应对突发
- **令牌桶：** 固定速率生成令牌，允许一定突发（桶容量）
- **滑动窗口：** 记录每个请求时间戳，滑动统计时间窗口内的请求数

**Redis 实现滑动窗口：** 使用 Sorted Set，score 为时间戳，每次请求时删除窗口外的记录，统计窗口内数量

### Q31：滑动窗口结构体字段设计？

**答：** 包含：窗口大小（windowSize）、请求计数器（counter）、时间戳队列、限流阈值（threshold）

## 十、基础算法与数据结构

### Q32：布隆过滤器是什么？如何控制误判率？

**答：**

- **原理：** 一个位数组 + 多个哈希函数；插入时将各哈希值对应位置为 1；查询时所有位置都为 1 则"可能存在"，任一位为 0 则"一定不存在"
- **问题：** 存在误判（False Positive），可能把不存在的元素判定为存在
- **控制误判率：** 增加位数组大小和哈希函数数量，计算公式 p ≈ (1 - e^(-kn/m))^k

### Q33：数据库索引失效的情况？

**答：**

- 对索引列使用函数或表达式（WHERE YEAR(date) = 2026）
- 隐式类型转换（WHERE phone = 13800000000，phone 为 varchar）
- LIKE 以 % 开头（LIKE '%keyword'）
- OR 条件中部分列无索引
- 联合索引不满足最左前缀原则
- 使用 !=、<>、NOT IN 等否定操作符

## 十一、RAG 性能优化与系统设计

### Q34：RAG 性能如何提升？

**答：**

- **检索层：** 向量索引优化（HNSW、PQ 量化）、结果缓存热点查询、并行检索
- **模型层：** 流式输出、KV-Cache、Prompt 缓存
- **系统架构：** 异步化非关键路径、预检索/预加载、CDN 加速静态资源

### Q35：大规模数据处理系统设计（求和场景）？

**答：**

- **千条级：** 单机内存直接处理
- **百万级：** MapReduce 分片，多 Worker 并行处理，最终汇总
- **保证效率与稳定：** 任务分片 + 重试机制 + 幂等设计 + 进度监控

## 十二、LLM 底层原理（基础八股）

### Q36：LLM 的底层原理？输入给模型的是什么？

**答：**

- 输入是 Token 序列，经过 Tokenizer 将文本转为 Token IDs
- 模型 = Token Embedding + 多层 Transformer Decoder
- 自回归生成：每次预测下一个 Token，拼接到已有序列后继续

### Q37：Self Attention 怎么实现？为什么分 QKV？

**答：**

- Q（Query）：当前 token 想要查询什么
- K（Key）：每个 token 能提供什么信息
- V（Value）：每个 token 实际携带的信息
- 计算：Attention(Q,K,V) = softmax(QK^T / √d_k) × V
- Q 和 K 计算相似度（注意力权重），权重加权 V 得到输出
- 同一个 token 的 Q/K/V 向量是不同的（经过不同的线性变换）

### Q38：Python 多线程与 GIL？

**答：**

- Python 有 GIL（全局解释器锁），同一时刻只有一个线程执行 Python 字节码
- GIL 存在原因：CPython 内存管理（引用计数）不是线程安全的
- Lock vs RLock：Lock 不可重入（同一线程重复 acquire 会死锁），RLock 可重入（同一线程可多次 acquire）
- CPU 密集型用多进程（绕过 GIL），IO 密集型用多线程

### Q39：信号量底层实现？

**答：** 本质上是一个计数器 + 等待队列：

- P 操作（acquire）：计数器减 1，若为负则线程阻塞
- V 操作（release）：计数器加 1，若 ≤ 0 则唤醒等待线程
- 底层依赖操作系统提供的原子操作（如 futex、信号量原语）

### Q40：C++ 编译链接流程？

**答：** 四个阶段：

1. **预处理：** 展开宏、处理 #include、条件编译
2. **编译：** 源码 → 汇编代码（词法分析、语法分析、语义分析、优化、代码生成）
3. **汇编：** 汇编代码 → 目标文件（.o/.obj）
4. **链接：** 合并目标文件和库，解析符号引用，生成可执行文件

## 十三、项目实践经验（高频综合题）

### Q41：大模型的工程实践经验有哪些方向？

**答：** 面试官可能从这些方向追问：

- **Prompt 工程：** 模板设计、Few-shot 策略、Chain-of-Thought 引导
- **上下文工程：** 信息压缩、优先级排序、To-Do List 引导
- **Fallback 策略：** 模型输出不可用时的兜底方案
- **Retry 机制：** 重试次数、退避策略、何时放弃
- **Reflection 机制：** 自我校验、输出修正循环
- **LLM 选型：** 成本 vs 效果、专项能力 vs 通用能力、私有化 vs API

### Q42：意图识别指标如何评测？

**答：**

- 离线评测：标注测试集 → 准确率/召回率/F1/混淆矩阵
- 在线评测：A/B 测试对比转化率、用户满意度
- 长期监控：意图分布漂移检测、badcase 闭环
