# 基于 RAG 的小红书知识库自动构建与智能检索 MCP — 面试问答

***

## 一、项目整体理解

### Q: 请用一句话概括这个项目解决了什么问题？

**答：** 小红书是封闭内容平台，外部 AI 工具无法直接检索其中的知识。本项目通过 Playwright 自动化采集 + PaddleOCR/EasyOCR 图文识别 + DeepSeek LLM 清洗分类，将小红书帖子转化为结构化 Markdown 知识库，再通过 ChromaDB 向量索引 + jieba 关键词检索 + 混合计分排序，最后封装为标准 MCP Server 暴露 `search_xhs`、`run_pipeline`、`search_kb` 三个 Tool，使 Claude Code 等 agent 可通过自然语言一键触发完整采集与检索链路。

### Q: 项目的核心价值是什么？

**答：** 打通三个闭环：

1. **内容获取** — Playwright 以 `launch_persistent_context` 复用登录态，绕过反爬与登录墙；OCR 补齐图文内容盲区
2. **知识沉淀** — LLM 去噪、修正 OCR 错字、分类打标，产出 Frontmatter + 正文的标准 Markdown，再按 `##` 标题切 Chunk 向量化入库
3. **生态集成** — MCP stdio 协议，一次开发即可被 Claude Code / Claude Desktop / 本地 Agent 等任意 MCP 客户端调用

## 二、RAG 架构与原理

### Q: 什么是 RAG？为什么选择 RAG 而不是微调？

**答：**

- **RAG（Retrieval-Augmented Generation）** 在 LLM 生成回答前，先从外部知识库中检索相关文档片段，以检索结果约束模型输出，降低幻觉。
- **选 RAG 而非微调的原因：**
  1. **知识实时更新** — 小红书内容频繁变化，RAG 只需重新采集、重新 index 即可
  2. **可追溯** — 检索结果明确展示引用文档路径和原文，微调是黑盒
  3. **零训练成本** — 不需要 GPU 微调，只维护本地 ChromaDB
  4. **幻觉抑制** — 检索到的文档直接将事实注入 Prompt，输出受约束

### Q: 讲一下 RAG 的核心流程

**答：** RAG（检索增强生成）核心分为三个阶段：

**1. 检索（Retrieval）**

- 将用户问题通过 Embedding 模型转为向量
- 在向量数据库中搜索最相似的 Top-K 文档片段
- 常用相似度度量：余弦相似度（cosine）、点积（dot product）

**2. 增强（Augmentation）**

- 将检索到的相关文档与用户问题组合
- 构建包含上下文的 Prompt，通常格式为：`[上下文] + [用户问题] + [回答要求]`

**3. 生成（Generation）**

- 将增强后的完整 Prompt 发送给 LLM
- LLM 基于检索到的真实知识生成答案，而非仅依赖模型自身知识

**简化流程：**

```
用户查询 → 向量化 → 向量检索 → 召回文档 → 拼装Prompt → LLM生成 → 返回答案
```

**项目中的实际实现：**

```
关键词 → Playwright 搜索页 → 逐个点击 note card → 提取 detail panel 文本
    → 解析 author/title/body/tags/日期/互动数 → 下载内容图片
    → PaddleOCR（回退 EasyOCR）识别图文 → 过滤小图/重复图 → overlap 检测合并
    → DeepSeek(OpenAI SDK) 格式化：合并 OCR 文本、修正错字、加标题层级
    → DeepSeek LLM 分类：按 categories.yaml 体系归类 + 摘要 + 关键词 + 情感/质量评分
    → 按分类生成 MD 文件 + _index.md + INDEX.md → 按 ## 标题切 Chunk 向量化入库
```

***

### Q: 整个 RAG 的数据流是怎样的？

**答：** 检索流程：

```
用户自然语言 query
    ├─→ jieba 分词 → 关键词匹配 frontmatter 中 tags/keywords 字段 → score = 命中词数/总词数
    └─→ ModelScope GTE-small Embedding → ChromaDB.query() → cosine distance → similarity = 1/(1+distance)
                                    ↓
             两路结果按 path 去重合并 → keyword-only 取自身分，semantic-only 取自身分，
             同时命中的 avg(两路分) 标记为 hybrid → 最终按 score 降序取 top_k
```

离线建库流程：

```
关键词 → Playwright 搜索页 → 逐个点击 note card → 提取 detail panel 文本
    → 解析 author/title/body/tags/日期/互动数 → 下载内容图片
    → PaddleOCR（回退 EasyOCR）识别图文 → 过滤小图/重复图 → overlap 检测合并
    → DeepSeek(OpenAI SDK) 格式化：合并 OCR 文本、修正错字、加标题层级
    → DeepSeek LLM 分类：按 categories.yaml 体系归类 + 摘要 + 关键词 + 情感/质量评分
    → 按分类生成 MD 文件 + _index.md + INDEX.md → 按 ## 标题切 Chunk 向量化入库
```


### Q: 除了当前的 RAG，业界还有哪些更先进或更复杂的 RAG 技术？

**答：**

#### 1. 成熟框架

| 框架 | 特点 |
|------|------|
| LangChain / LlamaIndex | 全套 RAG 工具链，支持复杂 Pipeline |
| RAGAS | 专门评估 RAG 效果的评测框架 |
| Haystack | 模块化 RAG 框架，支持多种检索器 |
| Dify | 可视化 RAG 编排，国内开源流行 |

#### 2. 高级检索策略

Query 改写和扩展方面，HyDE（Hypothetical Document Embeddings）让 LLM 先根据 query 生成假设答案，用这个答案去检索比直接用 query 效果更好；Query Decomposition 把复杂问题拆成多个子问题，分别检索再合并。

迭代检索方面，Self-RAG 让模型自己判断是否需要检索，反复迭代直到满意；ReAct RAG 结合推理（Reasoning）和行动（Acting），边想边搜。

多跳检索方面，CRAG（Corrective RAG）在检索结果差时触发修正，重新检索或降级到 Web 搜索。

#### 3. RAG + 生成优化

上下文压缩方面，LLMContextCompress 在送入 LLM 前先用一个小模型压缩上下文，去掉无关内容。答案生成方面，FuseChat RAG 多篇检索结果融合生成而不是逐篇拼接。Chain of Thought RAG 生成答案时分步骤推理，每步引用检索到的证据。

#### 4. 向量加关键词混合增强

| 技术 | 说明 |
|------|------|
| Hybrid Search + RR | 两路召回结果用 Reciprocal Rank 融合 |
| Sparse Embedding | 用 BGE-M3 等模型同时输出稀疏向量，提升关键词能力 |
| ColBERT | 延迟交互，只在最后做向量匹配，平衡精度和速度 |

#### 5. 生产级增强

路由机制是 query 进来后先判断属于哪个领域或任务，分发到不同的知识库或 Prompt 模板。缓存加预检索是相同 query 只检索一次，结果缓存复用。知识图谱增强方面 Graph RAG 先把文档抽成知识图谱，检索时不仅召回实体还召回关联关系，答案生成时结合图谱结构推理回答更全面。

#### 6. 最新前沿

| 技术 | 说明 |
|------|------|
| Agentic RAG | RAG 作为 Agent 工具，Agent 自己决定什么时候搜、搜什么、怎么用 |
| Adaptive RAG | 根据 query 复杂度自动选择策略，简单问题直接答复杂问题才检索 |
| Memo RAG | 在记忆模块中缓存中间结果，支持跨对话推理 |
| LightRAG | 轻量级图结构 RAG，兼顾简单和高效 |

#### 7. 与本项目的对比

| 维度 | 本项目 | 先进 RAG |
|------|--------|---------|
| 检索 | 两路混合 | HyDE / Self-RAG / CRAG |
| 迭代 | 无 | 多轮迭代修正 |
| 生成 | 无（只检索） | 结合 CoT / 图谱推理 |
| 上下文 | 全量送入 | 压缩/摘要/选择性送入 |
| 架构 | 单 Pipeline | Agent 化/可编排 |

### Q: 检索前可以对用户 query 做哪些改写？

**答：**

Query 改写是在检索之前先用 LLM 对用户 query 进行优化，让它变得更适合检索。为什么要改写呢，因为用户的问题往往是口语化、模糊或包含省略的，比如原始 query 说"那个字节的面试问啥"，改写后变成"字节跳动面试问题"。

常见改写策略有四种。HyDE 让 LLM 先根据 query 生成一个假设答案，用假设答案去检索效果比直接用 query 更好。Query Decomposition 把复杂问题拆成多个子问题分别检索再合并。Query Expansion 将一个 query 扩展成多个相似 query 召回更多结果。Query Correction 纠正拼写错误和口语化表达。

本项目当前没有 query 改写这一步，直接用原始 query 去检索。改进方向是加一层 LLM 调用把用户 query 改写成更规范更适合检索的表述再去 ChromaDB 检索。

## 三、5 阶段自动化流水线

### Q: 详细介绍一下流水线的各个阶段？

**答：** 流水线实际为 **5 个阶段**（OCR 内嵌在 Scrape 阶段中，不是独立阶段）：

| 阶段 | 模块           | 输入                | 输出                        | 关键技术                                                    |
| -- | ------------ | ----------------- | ------------------------- | ------------------------------------------------------- |
| 1  | Search       | 关键词               | SearchResult 列表           | `playwright.sync_api`，`launch_persistent_context` 复用登录态 |
| 2  | Scrape(+OCR) | SearchResult 列表   | XHSPost 列表(含 OCR 文本)      | 点击 note card → 提取 detail panel → 下载图片 → OCR             |
| 3  | Format       | XHSPost 列表        | 格式化后的 XHSPost             | DeepSeek API (OpenAI SDK) 清洗内容                          |
| 4  | Classify     | XHSPost 列表        | ClassifiedPost 列表         | DeepSeek LLM 按 YAML 分类体系归类                              |
| 5  | Build        | ClassifiedPost 列表 | Markdown KB + ChromaDB 索引 | 生成分类目录 MD 文件 + 向量入库                                     |

Search 阶段核心流程（[src/search/searcher.py](src/search/searcher.py)）：

- 拼接搜索 URL → 启动持久化浏览器 context → 检测是否触发验证码/限流/登录
- 滚动加载更多卡片 → 逐个解析 `section.note-item` 提取 title/author/cover\_url/post\_id

Scrape 阶段核心流程（[src/scrape/scraper.py](src/scrape/scraper.py)）：

- 搜索页逐卡点击 → `_extract_note_detail()` 提取 `#noteContainer` 内的文本
- `_parse_note_text()` 按 3 种已知结构解析 author/title/body/tags/日期/互动数
- 提取 carousel 内图片（排除 avatar/缩略图）→ 下载到 `output/knowledge_base/images/`
- **内联 OCR**：每张内容图片调 `ocr_images()` → 文字写入 `post.ocr_text`

### Q: 为什么需要 OCR？直接取 HTML 文本不行吗？

**答：** 小红书大量内容以**长图、截图、信息图**形式发布，文字以像素存在于图片中，DOM 里没有。PaddleOCR 专门针对中文优化（`lang="ch"`, `use_angle_cls=True`），配合 EasyOCR 回退，确保即使 PaddleOCR 环境有问题也能降级运行。

实际 OCR 流程（[src/scrape/ocr.py](src/scrape/ocr.py)）：

1. **图片过滤** — 跳过 <10KB 或 <200x200px 的图片（排除头像/icon）
2. **去重** — 按文件大小 hash 跳过重复图
3. **双重后端** — `_init_paddleocr()` 失败则回退 `_init_easyocr()`（`["ch_sim", "en"]`）
4. **重叠检测合并** — 多张图的 OCR 文本通过 `_has_overlap()` 检测末尾/开头的重复片段（最少 15 字重叠），merge 时去掉重复部分

### Q: 为什么要用 DeepSeek API 做内容清洗和分类？

**答：**

1. **OCR 纠错** — Format Prompt 明确要求修复 OCR 错误（如 "GlL"→"GIL"、"selfatention"→"self attention"），合并被错误断开的行
2. **格式标准化** — 识别面试题格式（"1. 2. 3." 或 "Q: A:" 结构），添加 `##` / `###` 层级，分离正文与评论区
3. **信息不增** — Prompt 要求"保持原始信息完整，不要添加原文中没有的内容"
4. **分类打标** — 按 `config/categories.yaml` 定义的 6 大分类体系（技术编程、产品设计、商业分析、职业发展、工具资源、生活方式）归类，输出 JSON 含 category/sub\_category/summary/keywords/entities/sentiment/quality\_score
5. **Rule-based 兜底** — 内容短于 100 字时跳过 LLM，直接用规则分类：`quality_score = length_score * 0.6 + interaction_score * 0.4`

### Q: 从获取小红书图片内容到存入本地 Markdown，哪些环节用 LLM，哪些不用？

**答：**

流程中各环节使用的技术如下：页面采集使用 Playwright 不需要 LLM，图片 OCR 使用 PaddleOCR 或 EasyOCR 不需要 LLM，内容清洗和纠错使用 DeepSeek API 需要 LLM，分类打标使用 DeepSeek API 需要 LLM，关键词提取使用 DeepSeek API 需要 LLM（嵌入在分类步骤中），Markdown 生成使用 DeepSeek API 需要 LLM，Chunk 切分使用正则表达式按 ## 标题切分不需要 LLM，向量嵌入使用 GTE-small 模型不需要 DeepSeek。

用 LLM 的环节主要是负责理解内容的部分，包括清洗修正 OCR 错误、分类打标、按 categories.yaml 体系归类，以及在分类步骤中同时提取 keywords 字段。不用 LLM 的环节是纯机械操作，包括采集用 Playwright 自动化、OCR 用 PaddleOCR 或 EasyOCR 模型处理图片文字识别、Chunk 切分用正则按 Markdown 二级标题切分、向量嵌入用 ModelScope 的 GTE-small sentence transformer 模型。一句话总结是 LLM 主要负责理解内容的环节，不需要 LLM 的环节是纯机械操作。

## 四、反爬与数据采集

### Q: Playwright 怎么绕过反爬与登录墙？

**答：** 具体措施（[src/search/searcher.py](src/search/searcher.py#L194-L209) 和 [src/scrape/scraper.py](src/scrape/scraper.py#L534-L542)）：

1. **持久化登录态** — `launch_persistent_context(user_data_dir="~/.xhs_browser_profile")`，所有 Cookie/localStorage 保留在磁盘，首次手动扫码后长期复用
2. **Chrome channel** — 使用系统安装的真实 Chrome（`channel="chrome"`），而非 Playwright 自带 Chromium，指纹更接近真实用户
3. **反检测启动参数** — `--disable-blink-features=AutomationControlled` 移除 `navigator.webdriver` 标识
4. **自定义 UA** — `Mozilla/5.0 ... Chrome/125.0.0.0 Safari/537.36`
5. **随机延迟** — 搜索间隔 0.5-2s，抓取间隔 3-8s（可配置 `scrape_min_delay`/`scrape_max_delay`），多关键词之间 5-10s 间隔
6. **验证码分级处理** — `_detect_block()` 检测 3 种异常状态：
   - `captcha_detected`（页面含"安全验证"或"滑块"）→ headless 模式抛 XHSRateLimitError；非 headless 模式等待 120s 手动解决
   - `rate_limited`（"访问频繁"）→ 抛 XHSRateLimitError 提示稍后重试
   - `login_required`（页面标题含"登录"）→ 抛 XHSLoginRequiredError 提示手动登录

### Q: 小红书的反爬机制有哪些？你们如何应对？

**答：**

- **X-s 签名** — 小红书 API 请求需要动态 X-s 签名，是逆向工程中最难的部分。本项目**不走 API**，通过 Playwright 控制真实浏览器走 DOM 渲染路径，完全规避了签名逆向问题
- **登录校验** — 未登录无法搜索。通过 `launch_persistent_context` 持久化 `~/.xhs_browser_profile` 目录，一次扫码长期有效
- **频率限制** — 间隔 3-8s 随机延迟，多关键词之间额外 5-10s 冷却
- **设备指纹** — 使用系统 Chrome + 真实 UA + `--disable-blink-features=AutomationControlled`

## 五、ChromaDB 与向量检索

### Q: 为什么选择 ChromaDB 而不是 Milvus/Weaviate/Pinecone？

**答：**

| 因素    | ChromaDB                                   | Milvus      | Pinecone |
| ----- | ------------------------------------------ | ----------- | -------- |
| 部署复杂度 | `pip install chromadb` 即用                  | 需 Docker/集群 | 云服务      |
| 场景    | 本地个人知识库（<100万条）                            | 企业级海量数据     | 生产级 SaaS |
| 开发体验  | `PersistentClient` + `get_collection` 语义清晰 | 功能丰富但需额外学习  | 网络调用     |
| 成本    | 免费                                         | 免费但运维重      | 按量付费     |

实际使用方式（[src/kb\_agent/indexer.py](src/kb_agent/indexer.py)）：

```python
client = chromadb.PersistentClient(path="output/chroma_db")
collection = client.create_collection(name="xhs_knowledge")
collection.add(ids=[chunk_id], embeddings=[embedding], metadatas=[{...}], documents=[chunk])
```

### Q: Embedding 模型具体是什么？为什么选它？

**答：** 使用 ModelScope 的 **`iic/nlp_gte_sentence-embedding_chinese-small`**（GTE 系列中文小模型），通过 `sentence_transformers.SentenceTransformer` 加载。

**选择原因：**

1. **中文原生优化** — GTE 系列是阿里在中文语义相似度任务上训练的，比 `all-MiniLM-L6-v2` 等通用英文模型在中文场景准确得多
2. **512 维** — 维度适中，存储开销和检索速度均衡
3. **本地运行** — 不走 Embedding API，无网络延迟，无调用成本
4. **ModelScope 下载** — 用 `modelscope.snapshot_download` 从国内镜像拉取，避免 HuggingFace 网络问题

实际加载代码（[src/kb\_agent/indexer.py](src/kb_agent/indexer.py#L23-L37)）：

```python
from modelscope import snapshot_download
from sentence_transformers import SentenceTransformer

model_dir = snapshot_download(
    "iic/nlp_gte_sentence-embedding_chinese-small",
    revision="master",
)
_embedder = SentenceTransformer(model_dir)
```

MCP Server 启动时调用 `_get_embedder()` 预加载模型到内存，避免首次检索时的冷启动延迟。
### Q: 向量化是怎么做的？

**答：**

向量化整体流程为文本经过 GTE-small 模型输出 512 维向量，然后存入 ChromaDB 存储。

具体步骤分为三步。第一步是模型加载，使用 ModelScope 的 GTE-small 中文模型，通过国内镜像下载避免 HuggingFace 网络问题。加载代码为 from modelscope import snapshot_download 和 from sentence_transformers import SentenceTransformer，通过 snapshot_download 下载模型后用 SentenceTransformer 加载。第二步是向量生成，单条文本调用一次 encode 方法输出 512 维浮点向量，再通过 tolist() 转为 Python 列表。第三步是批量入库，每个 Chunk 单独生成向量，和 metadata 一起 add 到 ChromaDB，collection.add() 包含 ids、embeddings、metadatas 和 documents 四个字段。

检索时的向量化同样使用 GTE-small 将用户 query 转为向量，再去 ChromaDB 做相似度搜索，调用 collection.query() 时传入 query_embeddings 和 n_results 参数。

技术选型原因如下：

| 设计 | 选择 | 原因 |
|------|------|------|
| 模型 | GTE-small | 中文语义优化，本地运行无 API 成本 |
| 下载源 | ModelScope | 国内镜像速度快，避免 HF 网络问题 |
| 向量维度 | 512 维 | 适中，存储和检索速度均衡 |



### Q: Chunk 是怎么划分的？

**答：** 按 **Markdown** **`##`** **二级标题**切分（[src/kb\_agent/indexer.py](src/kb_agent/indexer.py#L62-L71)）：

```python
sections = re.split(r"\n(?=## )", body)
for section in sections:
    section = section.strip()
    if section:
        chunks.append(section)
```

每个帖子生成一个 Markdown 文件，其章节结构为 `# 标题` / `## 摘要` / `## 正文` / `## 图片提取文字` / `## 图片` / `## 关键信息` / `## 原文链接`。`## 正文` 是最长的 Chunk，也是语义检索的主要匹配对象。如果正文无 `##` 子标题则将全文作为一个 Chunk。检索时 ChromaDB 返回最相似的 K 个 Chunk（含 metadata: title/category/path/url/tags/keywords）。

### Q: RAG 知识源的格式是什么样的？

**答：**

RAG 知识源采用 Markdown 文件格式存储，每个帖子生成一个 MD 文件，包含 frontmatter 元数据和正文内容两部分。

Frontmatter 位于文件开头，使用 YAML 格式定义元数据，包括 title（帖子标题）、category（所属分类）、sub_category（子分类）、tags（标签列表）、keywords（关键词，用逗号分隔）、summary（摘要）、url（原文链接）、likes/collects/comments（互动数据）、date（发布日期）等字段。

正文内容按 ## 二级标题组织结构，主要章节包括摘要（帖子主要内容概述）、正文（面试经验或内容详情）、图片提取文字（OCR 识别的图片内文字）、关键信息（公司、岗位、时间等关键字段）、原文链接（跳转回原帖的 URL）。

按 ## 二级标题切分后，每个 Chunk 的向量入库结构包含 chunk_id（如文件名_序号格式）、chunk_text（标题加内容的完整文本）、metadata（title、category、path、url、tags、keywords 等）。存储位置方面，Markdown 文件存放在 output/knowledge_base/categories/分类名/ 目录下，图片文件存放在 output/knowledge_base/images/ 目录下，向量索引存放在 output/chroma_db/ 目录下。

### Q: 知识更新怎么做的？底表里面会有哪些元数据？内容需要更新时底表怎么更新？

**答：**

当前实现采用全量重建模式，每次 build_index() 时先删除旧 collection 再重新创建，遍历所有 MD 文件重新向量化入库。

底表元数据包含 title（帖子标题）、category（所属分类）、path（MD 文件路径）、url（原文链接）、tags（标签，逗号分隔字符串）、keywords（关键词）、chunk_index（Chunk 序号）共七个字段。

内容更新时的处理流程是：新增帖子走 run_pipeline 采集生成新 MD 文件，然后调用 build_index 时自动 delete_collection 重建所有索引，解析全部 MD 文件按 ## 切 Chunk 后重新向量化入库。

当前实现存在局限性，每次全量重建数据量大时耗时长，新增一篇帖子也需要重建整个索引，无法针对单篇帖子做增量更新。

改进方向包括三个方面：增量 upsert 方案是新增帖子只 add 新 chunk 不删旧 collection；基于 path 去重的方案是 upsert 时按 path 判断是否已存在、已存在则更新；定时增量同步方案是比对 MD 文件变更只处理有变化的文件。

## 六、混合检索与排序融合

### Q: 混合检索是怎么做的？用了 RRF 吗？

**答：** 实际实现（[src/kb\_agent/rag\_engine.py](src/kb_agent/rag_engine.py#L142-L165)）是**两路独立检索 + 去重 + 简单计分平均**，并非标准 RRF：

```python
def hybrid_search(query, top_k=10):
    kw_results = keyword_search(query, top_k)    # jieba 分词匹配 frontmatter
    sem_results = semantic_search(query, top_k)   # ChromaDB 向量相似度

    merged = {}
    for r in kw_results:
        merged[r["path"]] = dict(r)              # 关键词独有结果
    for r in sem_results:
        key = r["path"]
        if key in merged:
            # 两路都命中：取平均分，标记 hybrid
            merged[key]["score"] = round((merged[key]["score"] + r["score"]) / 2, 3)
            merged[key]["method"] = "hybrid"
        else:
            merged[key] = dict(r)                 # 语义独有结果

    return sorted(merged.values(), key=lambda x: -x["score"])[:top_k]
```

**为什么没做标准 RRF：** 标准 RRF 需要排名信息（`1/(k + rank)`），而本项目两路检索的分数量纲不同——关键词分是 `[0, 1]` 的命中率，语义分是 cosine 转换后的 `[0, 1]` 相似度。两者量纲相近，直接平均即可合理融合，额外引入排名转换反而增加计算开销。分数量纲相近是本方案采用简单平均的前提。

### Q: 召回和重排分别用了什么算法？

**答：**

**召回（Recall）阶段 — 两路独立检索：**

| 召回路   | 算法                                          | 实现                                                              |
| ----- | ------------------------------------------- | --------------------------------------------------------------- |
| 关键词召回 | **jieba 分词 + 精确匹配**                         | 对 query 分词后，匹配 frontmatter 的 tags/keywords 字段，命中即得分             |
| 语义召回  | **GTE-small Embedding + Cosine Similarity** | ModelScope GTE-small 向量化，ChromaDB query 返回 cosine distance 转相似度 |

**重排（Rerank）阶段 — 项目中未使用专门的 rerank 模型**

项目只做了简单的**分数融合**，不是标准 rerank：

- 两路独立召回后按 path 去重
- 只有一路命中：保留原分数
- 两路同时命中：avg(关键词分, 语义分) 作为最终分数
- 最终按分数降序排列返回 top\_k

**未采用 RRF / Cross-Encoder 的原因：**

- 两路分数量纲都是 \[0, 1]，量纲相同，直接平均即可
- Cross-Encoder rerank 虽效果更好，但需要额外调用 LLM 或 rerank 模型，增加延迟和成本
- 当前阶段优先级是快速验证，rerank 可作为后续优化点

**后续优化方向：** 可引入 BGE-Reranker 等 Cross-Encoder 模型对召回结果做 rerank，进一步提升排序效果。

### Q: ChromaDB 具体在这个项目中是如何使用的？

**答：**

**建库阶段：** 解析 Markdown 文件，按 ## 二级标题切分成多个 Chunk，每个 Chunk 调用 ModelScope 的 GTE-small 模型生成 512 维向量，然后通过 PersistentClient 创建本地持久化 Collection，将 chunk\_id、向量、metadata（title/category/path/tags/keywords 等）和原文文本一起 add 进去，数据存储在 output/chroma\_db/ 目录。

**检索阶段：** 用户 query 先用同一个 GTE-small 模型转为向量，然后调用 collection.query() 在 ChromaDB 中做向量相似度搜索，返回 Top-K 个最相似的 Chunk。ChromaDB 内部使用 cosine distance 计算相似度（0 表示完全相同，2 表示完全相反），通过 1/(1+distance) 转换为 0\~1 的相似度分数。

**分数融合：** 语义检索返回的分数与关键词检索的分数在 Python 层做合并，两路都命中的取平均分，只有一路命中的保留各自分数，最终按分数降序返回。

**持久化：** 使用 PersistentClient 而非 Client，所有向量和 metadata 落本地磁盘的 SQLite 文件，进程重启后数据保留，不需要重新建库。

**局限性：** 当前每次 build\_index 会 delete\_collection 再 create\_collection，属于全量重建模式，不支持增量。万级以上数据量可考虑 upsert 或迁移到 Milvus/Qdrant。

### Q: ChromaDB 说"所有向量和 metadata 落本地 SQLite 文件"这句话如何理解？

**答：** 这个表述不够准确。ChromaDB 的 PersistentClient 在本地 output/chroma\_db/ 目录下会生成多个文件，包括 DuckDB（存储向量数据和 HNSW 索引）和 SQLite（存储 collection 和文档的 metadata）。DuckDB 是列式存储引擎，专为向量运算优化；SQLite 负责结构化 metadata。两者都是嵌入式本地存储，不需要启动独立服务进程。

### Q: ChromaDB 与独立 SQLite 有什么区别？

**答：** 独立 SQLite 是通用关系型数据库，需要写 SQL 操作表结构，手动实现向量相似度计算。ChromaDB 是面向向量检索优化的数据库，底层集成 DuckDB（向量计算）+ SQLite（metadata）+ HNSW（向量索引），对外提供 add/query 接口，自动处理向量存储、相似度计算和 Top-K 排序。本质上 ChromaDB 不是"用了 SQLite"，而是集成了多种嵌入式存储引擎提供统一的向量检索能力。

### Q: 有哪些简单的重排算法可以用于后续改进？

**答：**

#### 1. RRF（Reciprocal Rank Fusion）

**公式：**

```
score(doc) = Σ 1/(k + rank_i(doc))
```

- k 通常取 60
- rank\_i 是该文档在不同召回路中的排名

**优点：** 无需调参，只依赖排名信息，不依赖分数量纲
**适用场景：** 两路召回分数量纲不一致时

#### 2. Cosine 线性融合（项目当前方案）

**公式：**

```
score(doc) = α × keyword_score + (1-α) × semantic_score
```

- α 可调（项目默认 0.5，即平均）

**优点：** 简单，参数少
**缺点：** 需要分数量纲相近才合理

#### 3. BM25 召回 + 向量重排（两阶段方案）

**流程：**

```
Query → BM25 召回 Top-100 → Cross-Encoder rerank → Top-10
```

**核心思想：** 第一层粗召回靠 BM25 保证召回率，第二层 Cross-Encoder 精排保证相关性

#### 4. Cross-Encoder Rerank（效果最好）

**原理：** 将 query 和 document 一起输入 encoder，输出一个相关性分数

```
Cross-Encoder(query, doc) → score ∈ [0, 1]
```

**常用模型：**

| 模型                       | 规模   | 效果 |
| ------------------------ | ---- | -- |
| BAAI/bge-reranker-base   | 279M | 基准 |
| BAAI/bge-reranker-large  | 434M | 最好 |
| Alibaba-NLP/gte-reranker | 136M | 轻量 |

**优点：** 考虑 query 和 doc 的交互，精度高
**缺点：** 计算慢（不能做向量索引），适合小规模重排

#### 5. 商业 Rerank API

如果不想本地部署，可用 API 服务：

- **MiniMax Rerank API**
- **Cohere Rerank**
- **Jina Rerank**

输入 query + document list，输出排序结果。按调用次数计费，适合快速上线。

#### 改进建议（针对本项目）

**最低成本方案：** 在现有两路召回后，加一层 **RRF 融合** 替代简单平均，对两路召回的 Top-20 结果重排序，取 Top-10 返回。

**效果最优方案：** 引入 **BAAI/bge-reranker-base**，对 keyword 和 semantic 两路召回的并集（去重后约 Top-20）做 rerank，输出最终 Top-10。

### Q: 这个项目里的分片（Chunk）是如何实现的？

**答：**

分片策略是按 Markdown 的 ## 二级标题切分。使用正则 `re.split(r"\n(?=## )", body)` 匹配以 ## 开头的行，实现切分。

具体流程如下：第一步，读取 Markdown 文件，先解析 frontmatter 元数据（title/category/tags 等），找到正文起始位置。第二步，跳过 frontmatter 部分，对正文 body 按 ## 二级标题 split。第三步，每个 ## 标题下的内容作为一个 Chunk。第四步，如果正文没有任何 ## 标题，则将整篇正文作为一个 Chunk。

例如，一个帖子 MD 文件结构为 # 标题 / ## 摘要 / ## 正文 / ## 图片提取文字 / ## 关键信息，经过切分后得到多个 Chunk，如 Chunk 0 为 ## 摘要 下的内容，Chunk 1 为 ## 正文 下的内容，Chunk 2 为 ## 图片提取文字 下的内容。

每个 Chunk 单独生成向量和 chunk\_id（如 `字节一面面经_0`、`字节一面面经_1`），存入 ChromaDB 时 metadata 记录所属文章标题、分类、路径、标签等信息，便于后续检索结果关联回原文章。

### Q: 如果正文过长，这个 Chunk 策略合理吗？

**答：**

当前策略存在两个潜在问题。

问题一，Embedding 精度下降。GTE-small 这类 sentence transformer 模型有最大输入长度限制，通常是 512 个 token。如果 ## 正文 下的内容超过这个长度，文本会被截断，embedding 无法完整捕获全文语义，导致相似度计算不准确。

问题二，检索结果稀释。一个很长的 Chunk 可能包含多个主题。比如一篇面经的 ## 正文 写了 3000 字，涵盖了算法、系统设计、HR 提问三个部分。用户查询"系统设计"，整篇正文都会被检索出来，但大部分内容（算法部分）对用户是噪音，灌入 LLM 的上下文也不必要地冗长。

改进方案包括：固定长度分块加重叠，如每 500 字为一个 chunk，相邻 chunk 之间重叠 50 字，避免切断语义；语义分块，用 LLM 识别自然段落边界，在段落边界处切分，比固定长度更合理但成本更高；结合标题层级，如果 ## 正文 特别长，可以进一步按 ### 三级标题切分，或者在正文内部按固定长度二次切分。

当前项目的折中方案是，按 ## 二级标题切分已经是一个合理的折中，小红书帖子的正文通常不长（几百字到一两千字），LLM 上下文窗口也能容纳，不至于严重稀释语义。但如果是长文内容（如公众号文章），当前策略就需要改进。

### Q: 这个项目中，Chunk Size 和 TopK 的参数实验是如何设计与开展的？

**答：**

根据代码和配置分析，这个项目没有做系统的参数实验。

实际情况如下：top\_k 硬编码为默认值 10，调用方传入参数覆盖；chunk\_size 没有这个参数，分片完全由 ## 二级标题决定，无法人工干预 chunk 长度；项目中没有任何记录不同参数下检索效果的实验日志。

这些参数是如何确定的：top\_k=10 是基于经验设定的，返回 10 条结果足够用户浏览，又不至于让 LLM 上下文过于冗长。chunk 靠 ## 标题自然切分，也是跟着 Markdown 本身的结构走，没有额外控制。

如果要系统设计参数实验，可以参考以下方案：

| 参数           | 实验设计                              | 评测指标        |
| ------------ | --------------------------------- | ----------- |
| top\_k       | 取 5/10/20/50，分别测召回率和 MRR\@K       | MRR\@K、准确率  |
| chunk\_size  | 按字数 200/500/1000/2000 切分，测语义完整度   | 人工标注相关性、AUC |
| embedding 模型 | GTE-small vs text2vec-base vs m3e | NDCG\@10    |

结论是，项目处于快速验证阶段，参数基本靠经验值，没有做系统的 A/B 实验。如果要优化检索效果，第一步应该是建立标注数据集（Query-Document 相关性），然后才能量化不同参数配置下的实际改进。

### Q: 什么是召回率？MRR@K 又是什么？

**答：**

召回率（Recall）定义是在所有相关文档中，被检索系统成功召回的比例。计算公式为：召回率 = 检索到的相关文档数 / 所有相关文档总数。例如知识库里共有 100 篇相关文档，系统返回了 40 篇，则召回率 = 40%。

MRR\@K（Mean Reciprocal Rank @ K）是多个查询的平均倒数排名。对每个查询，找到第一个相关文档的位置 K，倒数排名 = 1/K，MRR\@K = 所有查询的倒数排名之和 / 查询总数。例如查询1第一个相关文档在第1位得1.0，查询2在第3位得0.333，查询3在第5位得0.2，MRR\@5 = (1.0 + 0.333 + 0.2) / 3 ≈ 0.511。MRR 只关注第一个相关文档的位置，适合搜索场景（用户只关心最靠前的结果够不够准）。

Recall\@K 和 MRR\@K 的区别在于：Recall\@K 关注尽量多召回相关文档，适用于知识库补全、推荐系统；MRR\@K 关注第一个结果要准，适用于搜索引擎、问答系统。

### Q: 如果要测试系统的召回率，如何进行设计实验并进行测试？

**答：**

测试 RAG 系统召回率的实验设计分为五个步骤。

第一步，构建标注数据集。准备 50-100 个真实用户 query，对每个 query 标注出知识库中所有相关文档（相关 = 能够回答这个问题）。标注原则是至少 3 名标注者独立标注后取交集，相关性分为 3 级：完全相关、部分相关、不相关。

第二步，构造评估脚本。伪代码示例如下：对待测 query 调用系统 retrieve 函数获取 top\_k 条结果，与标注数据集中的相关文档做交集运算，计算 recall = 命中相关文档数 / 所有相关文档数，遍历所有 query 求平均召回率。

第三步，设计对比实验。对照组 baseline 使用 top\_k=10 纯语义检索；实验组 1 扩大召回窗口到 top\_k=20；实验组 2 改用 hybrid 混合检索；实验组 3 在 hybrid 基础上改用 RRF 融合策略。实验设计表格如下：

| 实验组      | 配置                        | 变量          |
| -------- | ------------------------- | ----------- |
| baseline | top\_k=10, 纯语义检索          | 对照组         |
| 实验1      | top\_k=20, 纯语义检索          | 扩大召回窗口      |
| 实验2      | top\_k=10, hybrid检索       | 混合召回 vs 纯语义 |
| 实验3      | top\_k=10, hybrid检索 + RRF | 不同融合策略      |

第四步，计算评测指标。主要指标包括 Recall\@K（相关文档被召回的比例）、MRR\@K（第一个相关文档的排名质量）、NDCG\@K（综合相关性和排名的质量）。输出报告示例为实验名称列 Recall\@10、MRR\@10、NDCG\@10 四个实验组的对应数值。

第五步，分析 bad case。挑出召回失败的前 10 个 query，分析原因是 Embedding 模型语义理解偏差、Chunk 切分把相关段落切断了、还是知识库本身就没有相关文档。

关键注意事项：top\_k 要与实际使用场景一致（如果产品上默认返回 5 条，评估时也应测 5）；标注数据集要定期更新（用户 query 分布会变化）；区分系统不行和知识库缺失（如果知识库里没有相关内容，召回率天然就低）。

### Q: 查询时的 TopK 是如何做召回的？

**答：**

查询时 TopK 召回的整体流程是：用户 query 输入后并行执行两路召回（关键词 + 语义），结果合并去重，按分数排序，取 TopK 返回。

第一路关键词召回（keyword\_search）的流程如下：对 query 做 jieba 分词，过滤掉单字和空白；遍历知识库所有 MD 文件，读取 frontmatter 的 tags 和 keywords 字段；逐个匹配 query 中的词是否出现在 tags/keywords 里；命中词数除以 query 总词数得到关键词得分；按得分降序取 top\_k 条。

第二路语义召回（semantic\_search）的流程如下：用 GTE-small 将 query 转为 512 维向量；调用 ChromaDB 的 collection.query() 传入 query 向量；ChromaDB 内部用 HNSW 索引做向量相似度搜索；返回 cosine distance 最近的 top\_k 条；通过 1/(1+distance) 转换为相似度分数。

两路结果合并时，先按 path 或 key 去重。两路都命中的文档取两个分数的平均并标记为 hybrid，只有一路命中的保留各自分数，最终按分数降序取 top\_k 返回。

以 top\_k=3，query="字节算法面试" 为例：文档"字节一面.md"关键词分 0.67、语义分 0.89、最终分 0.78、来源 hybrid；文档"腾讯算法题.md"关键词分 0.33、语义分 0.72、最终分 0.53、来源 semantic；文档"字节二面.md"关键词分 0.67、语义分 0.31、最终分 0.49、来源 hybrid。返回前 3 条，结果已按最终分数降序排列。

### Q: 关键词检索具体是怎么匹配的？

**答：** 实际不是 BM25（[src/kb\_agent/rag\_engine.py](src/kb_agent/rag_engine.py#L49-L96)）：

```python
import jieba
query_kw = [k.lower() for k in jieba.cut(query) if k.strip() and len(k.strip()) > 1]
# 仅匹配 frontmatter 中的 tags 和 keywords 字段
match_text = f"{tags_str} {kw_str}"
score = sum(1 for k in query_kw if k in match_text)
# 最终分 = 命中词数 / 查询总词数
score = round(score / max(len(query_kw), 1), 2)
```

关键设计决策：**关键词检索只命中 frontmatter 元数据（tags + keywords），不扫全文正文**。这样做的原因是：(1) 分类阶段 LLM 已将核心概念提取到 tags/keywords 中，匹配元数据等价于匹配核心概念；(2) 大幅减少误匹配，避免"Python"命中一篇只提到了"Python"但主题无关的帖子。

### Q: BM25 和向量检索在双路混合检索中各自解决什么问题？

**答：**

两者在混合检索中各有分工，不是竞争关系，而是互补关系。

**BM25（稀疏检索）解决的问题：**

- **精确术语匹配** — 用户 query 中的专有名词、缩写、技术栈名称（如"Python"、"RAG"、"transformer"），BM25 能精准命中包含这些词的文章
- **短 query 友好** — 用户 query 短时（如"字节面试"），向量 embedding 难以从有限语义中推断意图，BM25 直接词匹配更可靠
- **可解释性强** — 用户问"为什么召回这篇"，BM25 可以说"因为包含3个关键词"，可解释可调
- **计算高效** — 倒排索引结构，CPU 友好，无需 GPU 算向量

**向量检索（稠密检索）解决的问题：**

- **语义相似** — 用户说"怎么准备算法面试"，没有文章标题包含这5个字，但向量检索能找到语义相关的内容
- **同义词泛化** — "计算机"和"电脑"、"面试"和"面经"，词形不同但语义相近，向量检索能跨过去
- **口语化 query** — 用户说"那个字节的面试问啥"，向量检索能理解真实意图，而 BM25 可能一个词都匹配不上
- **模糊意图** — 当 query 本身语义模糊时，向量检索找全局最优匹配

**两者融合的价值：**

双路混合检索 = BM25 的精确召回 + 向量检索的语义泛化。精确召回保证不漏掉明确匹配的文章，语义泛化保证不漏掉意思相近但表述不同的文章。

**本项目中的对应关系：**

本项目的关键词召回（jieba + 精确匹配）≈ 简化版 BM25，语义召回（GTE-small + ChromaDB cosine）= 向量检索。两者独立召回后合并，两路都命中则 avg 分数，兼顾精确性和语义性。

### Q: 三种检索模式的适用场景？

**答：**

| 检索模式     | 实现                                             | 适用场景                              |
| -------- | ---------------------------------------------- | --------------------------------- |
| keyword  | jieba 分词匹配 frontmatter                         | 精确查找已知概念、专有名词："transformer 注意力机制" |
| semantic | GTE-small Embedding → ChromaDB cosine distance | 模糊语义查询："如何准备算法面试"                 |
| hybrid   | keyword + semantic 去重合并，命中两路则 avg              | 通用场景（默认模式），兼顾精确与语义                |

默认使用 hybrid。调用方可指定 `layers=["keyword"]` 或 `layers=["semantic"]` 切换模式（[src/kb\_agent/searcher.py](src/kb_agent/searcher.py#L23-L74)）。

### Q: Rerank 的作用是什么？Rerank 模型和 Embedding 模型有什么区别？

**答：**

**Rerank 的核心作用：**

Rerank（重排）是检索流程的第二阶段，位于粗召回之后。Embedding 召回是"大海捞针"，在百万文档中快速找到 Top-100 相关文档；Rerank 是"精挑细选"，在这 100 篇中重新排序，最终输出 Top-10 最相关的。

**核心区别：**

| 维度 | Embedding 模型（Bi-Encoder） | Rerank 模型（Cross-Encoder） |
|------|------|------|
| **编码方式** | query 和 doc 分别独立编码为向量 | query 和 doc 一起输入 encoder |
| **交互时机** | 编码时无交互，相似度靠向量运算 | 编码时深度交互，考虑每个 token 对 |
| **速度** | 快，可预计算 doc 向量，实时算 query 向量 | 慢，每个 doc 都要单独 forward 一遍 |
| **精度** | 较高（召回阶段） | 更高（精排阶段） |
| **适用规模** | 海量文档（百万级） | 小规模候选集（百级） |

**模型层面详细对比：**

| 维度 | Bi-Encoder（Embedding） | Cross-Encoder（Rerank） |
|------|------|------|
| **网络结构** | 共享 Transformer encoder，query/doc 共享同一个 encoder | 共享 Transformer encoder，query/doc 一起通过 encoder |
| **输入方式** | `encode(query)` 和 `encode(doc)` 分开独立调用 | `[CLS] query [SEP] doc [SEP]` 拼接后一次 forward |
| **Attention** | query token 只看自己的上文，doc token 也只看自己的上文 | query token 可以看到 doc 所有 token，反之亦然（双向 attention） |
| **输出** | \[CLS\] 向量（或 mean pooling）→ 512/768 维向量 | \[CLS\] 向量 → MLP → sigmoid → 0\~1 分数 |
| **向量可缓存** | doc 向量可预计算并缓存，实时只算 query | 每个 (query, doc) pair 都要完整 forward，无法缓存 doc |
| **训练目标** | contrastive learning：拉近相似对、推远不相似对 | 二分类：相关=1，不相关=0，端到端训练 |

**Attention 机制的实质差异：**

```
Bi-Encoder（独立编码）：
  Query 向量计算 Attention 时：
    → 只attend到 query 自己的 tokens
    → 完全看不见 doc 的 tokens
  Doc 向量计算 Attention 时：
    → 只attend到 doc 自己的 tokens
    → 完全看不见 query 的 tokens
  
  结果：向量q 和 向量d 的编码过程彼此独立，无交互

Cross-Encoder（联合编码）：
  [CLS] q tokens... [SEP] d tokens... [SEP]
    → 每个 token 都能attend到序列中的所有其他 token
    → query 中的 "面试" 可以attend到 doc 中的 "字节跳动"
    → 捕获细粒度的词语对应关系（如 query 的 "算法" 对应 doc 的 "算法题"）
  
  结果：query 和 doc 在同一语义空间中深度交互
```

**信息损失的具体表现：**

Bi-Encoder 的本质缺陷是编码时 query 和 doc 互不可见。例如：

- query="字节算法面试"
- doc="今天去字节跳动了，面试官问了很多关于CNN和RNN的问题"

Bi-Encoder 编码时，"算法" 这个词在 doc 向量中只是一个静态语义表示（可能是"神经网络层类型"的上下文），无法捕获它恰好与 query 中"算法"完全一致这个事实。

Cross-Encoder 则能在 attention 层直接看到"算法"在 query 和 doc 中同时出现，精确捕获匹配关系。

**速度差异的根源：**

Bi-Encoder 只需 N+1 次 forward（N 篇 doc + 1 次 query），且 doc 向量可预先算好存进向量数据库。Cross-Encoder 对每个候选 doc 都要做一次完整 forward，若要对 100 篇 doc 做 rerank，需要 100 次完整的 Transformer forward，计算量是 Bi-Encoder 的 100 倍。因此 Cross-Encoder 只适合小规模精排，不适合大规模召回。

**典型流程（两阶段检索）：**

```
Query → Embedding 召回 Top-100 → Cross-Encoder 逐个打分 → 按分数重排 → Top-10
```

**常用 Rerank 模型：**

| 模型 | 规模 | 说明 |
|------|------|------|
| BAAI/bge-reranker-base | 279M | 基准模型 |
| BAAI/bge-reranker-large | 434M | 效果最好 |
| Alibaba-NLP/gte-reranker | 136M | 轻量版 |
| Cohere Rerank | API | 商业服务 |

**本项目当前状态：**

项目只做了分数融合（avg），没有专门的 Rerank 阶段。两路召回去重后直接按平均分排序返回，没有 Cross-Encoder 做精排。这是后续优化的重点方向。

### Q: 如何评测 RAG 的效果？

**答：**

RAG 效果评测分为三个层面。

检索阶段指标包括 Recall@K（相关文档被召回的比例，适用于知识库补全和推荐系统）、MRR@K（第一个相关文档的排名质量，适用于搜索引擎和问答系统）、NDCG@K（综合相关性和排名的质量，用于排序效果评估）、Hit@K（Top-K 中是否包含相关文档，用于快速判断召回能力）。

生成阶段指标包括 Answer Accuracy（答案相对标注答案的准确率）、RAGAS（基于检索内容和生成质量打分）、Hallucination Rate（幻觉率，生成内容与检索内容不符的比例）。

评测方法分为四步：第一步构建测试集，准备 50-100 个真实用户 query，对每个 query 标注相关文档，至少 3 人独立标注取交集作为 ground truth；第二步检索评测，query 经过系统检索后与 ground truth 对比计算 Recall、MRR、NDCG；第三步生成评测，query 加检索结果送入 LLM 生成答案后由人工或 LLM 评估答案质量；第四步端到端评测，用标注数据集测试以问答准确率作为最终指标，同时区分检索差和生成差来定位问题根因。

项目实际情况是，没有做系统的评测实验。如果要评测本项目，检索评测方向是标注测试集计算 Recall@10 和 MRR@10，生成评测方向是本项目不生成答案无需评测，改进优先级是先建标注数据集再量化改进效果。

### Q: RAG 返回了冲突的内容怎么办？

**答：**

问题场景是两篇检索结果对同一个问题给出了矛盾答案，例如文档A说 RAG 的 Embedding 用的是 BGE，文档B说用的是 M3E，用户问你们用的是什么 Embedding 模型时不知道该信哪个。

原因分析有三个：知识库内容过时（不同时间采集的帖子信息不一致）、来源不可靠（不同帖子作者认知不同）、Chunk 切分问题（关键信息被切断在不同 Chunk 导致上下文缺失）。

解决思路有四个方案。方案一是置信度排序，多个结果冲突时优先返回与用户 query 关键词最匹配的那篇或发布时间最新的那篇。方案二是来源标注，返回结果中明确标注来源让用户自己判断哪篇更可信。方案三是冲突检测加提示，如果发现两篇内容矛盾，在返回结果中标注以下内容存在冲突请参考原文判断，而不是强行融合。方案四是 LLM 判断，把多篇检索结果一起发给 LLM 让它判断是否存在冲突，并给出综合结论或指出冲突点。

项目改进方向是当前项目没有处理冲突的机制，可以加一个后处理步骤：检索结果按 score 排序后检测 content 中是否存在矛盾关键词（如两个不同的 Embedding 模型名），如果检测到冲突则在结果中标注内容冲突建议查看原文。

### Q: 两条政策知识冲突时如何判断置信度？取哪一条作为检索结果？

**答：**

问题本质是两条知识冲突需要判断哪条更可信，RAG 检索本质上是信息检索而不是推理判决，需要结合多种置信度信号综合判断。

可参考的置信度信号包括录入时间（新政策覆盖旧政策，优先返回最新的）、录入人权威度（管理员或官方账号高于普通员工）、来源标注（明确标注官方政策高于员工个人理解）、点赞和互动数（互动多的版本经过更多人验证）、知识状态（已审核高于待审核和草稿）、关键词命中（与 query 匹配度越高的越优先）。

具体策略有四个。策略一是时间优先，返回录入时间更近的那条，但需要确保知识库中新确实代表当前有效政策。策略二是权威来源优先，官方公告高于员工录入，管理员发布高于普通员工发布。策略三是交叉验证，query 问的是拆吊牌规则时返回提到拆吊牌关键字的那条优先。策略四是聚合多数，如果知识库中有多个版本的政策返回被引用次数最多的那个版本。

实际工程建议方面，做法包括加 metadata 字段（source_type、author、is_official、created_at）、查询时加权（按 source_type 和时间综合排序）、冲突检测（返回结果时标注以下存在冲突让用户判断）、知识治理（定期清理过期知识，官方知识优先覆盖）。

一句话总结是 RAG 本身不解决知识对错问题，置信度取决于 metadata 设计的丰富程度，建议在录入环节就区分官方政策和员工个人理解，查询时优先返回官方来源。

## 七、MCP 协议与服务封装

### Q: 什么是 MCP？为什么封装成 MCP Server？

**答：**

- **MCP（Model Context Protocol）** 是 Anthropic 提出的开放协议，基于 JSON-RPC 2.0，标准化了 AI 应用与外部工具/数据源的交互。核心概念：Tools（工具调用）、Resources（资源读取）、Prompts（提示模板）。
- **封装原因：**
  1. **一次开发，多端使用** — 符合 MCP 规范即可被 Claude Code、Claude Desktop、Cursor 等客户端调用
  2. **标准化接口** — Tool 的 inputSchema 用 JSON Schema 严格定义，LLM 可准确生成调用参数
  3. **自然语言驱动** — 用户说"搜索小红书关于XX的内容"，agent 自动调 `search_xhs` Tool

### Q: 如果检索为空是怎么处理的？

**答：**

检索为空时的处理分为三层。

第一层关键词召回为空时，如果两路召回都没有结果，直接返回空列表。代码实现为 if not kw\_results and not sem\_results: return \[]。

第二层语义召回为空时，如果 ChromaDB 返回空 ids，也返回空列表。代码实现为 if not results\["ids"] or not results\["ids"]\[0]: return \[]。

第三层格式化展示为空时，最终返回给用户时展示"未找到相关内容"的友好提示。代码实现为 if not results: return "## 检索结果: "{query}"

未找到相关内容。"。

总结表格如下：

| 层级  | 场景             | 处理方式          |
| --- | -------------- | ------------- |
| 召回层 | 两路都没结果         | 返回空列表 \[]     |
| 召回层 | ChromaDB 无 ids | 返回空列表 \[]     |
| 展示层 | 最终结果为空         | 返回"未找到相关内容"提示 |

项目没有额外降级策略（如回退到纯关键词、或返回随机内容），直接告知用户未找到相关内容。

### Q: 如果召回的内容和用户问题不相关，怎么避免？

**答：**

召回内容不相关的核心原因有三个：Embedding 模型语义理解偏差导致 query 和 doc 表面词不搭边但语义相关的情况没学好；Chunk 切分把相关段落切断了导致相关内容被截断在两个 chunk 里；知识库本身缺少相关内容导致系统尽力了但确实没有。

解决方案包括四个方向。方案一是设置相似度分数阈值，对低于 0.5 分的结果直接过滤掉，同时多召回一些（如 top\_k\*2）再过滤，保证最终数量。方案二是引入 Rerank 做二次筛选，用 Cross-Encoder（如 BGE-Reranker）重新评估 query 和 doc 的相关性，过滤掉表面相似但语义不相关的文档。方案三是多路召回交叉验证，只有同时被关键词和语义两路召回的文档（hybrid）才认为是高质量的，纯一路召回的结果可信度相对较低。方案四是 LLM 做答案质量判断，在返回给用户之前先判断召回内容能不能回答用户的问题，如果不相关则告知用户未找到相关内容。

当前项目可以改进的地方如下：

| 改进点         | 具体做法                            |
| ----------- | ------------------------------- |
| 加分数阈值       | 对 semantic 召回结果设 0.5-0.6 阈值过滤   |
| 引入 rerank   | 接入 BGE-Reranker 对 top20 做二次排序   |
| 标注 bad case | 分析低分但相关的文档，优化 Chunk 策略          |
| LLM 判断      | 在 hybrid\_search 后加一层 LLM 相关性判断 |

一句话总结：根本解法是分数阈值过滤加上 Rerank 二次筛选，再加上 bad case 分析迭代优化。不能完全依赖模型本身，要加规则护栏。

### Q: 对外暴露了哪些 Tool？参数和设计思路是什么？

**答：** （[mcp\_server/xhs\_server.py](mcp_server/xhs_server.py#L163-L223)）

| Tool           | 参数                          | 超时   | 功能                                     | 设计思路            |
| -------------- | --------------------------- | ---- | -------------------------------------- | --------------- |
| `search_xhs`   | `keyword`(必填), `count`(默认5) | 90s  | 搜索小红书返回标题/URL/作者列表                     | 轻量预览，不触发采集，快速验证 |
| `run_pipeline` | `keyword`(必填), `count`(默认5) | 600s | 完整 search→scrape→format→classify→build | 一键建库，适合深度研究     |
| `search_kb`    | `query`(必填), `top_k`(默认5)   | 30s  | 混合检索本地知识库返回文档+分数                       | 本地检索应快速响应       |

**渐进式设计：** `search_xhs`（看看有什么）→ `run_pipeline`（采下来）→ `search_kb`（检索已采集内容），三步渐进，用户按需选择。

**错误透传：** `run_pipeline` 和 `search_xhs` 捕获特定异常（XHSLoginRequiredError → 提示扫码登录，XHSRateLimitError → 提示等一等），不 crash，返回中文提示给用户。

### Q: MCP Server 的通信方式是什么？

**答：** 使用 **stdio（标准输入输出）**（[mcp\_server/xhs\_server.py](mcp_server/xhs_server.py#L283-L288)）：

```python
async with stdio_server() as (reader, writer):
    await server.run(reader, writer, server.create_initialization_options())
```

**关键细节：**

- 日志不走 stdout（会干扰 MCP 协议），单独写入 `output/mcp_server.log`
- 所有同步操作（Playwright/ChromaDB/Embedding）通过 `loop.run_in_executor(None, ...)` 放入线程池，避免阻塞 async event loop
- 启动时 `main()` 中预加载 Embedding 模型（30s 超时），失败不阻塞启动（非致命错误）
- 每个 Tool 调用记录耗时和结果长度到日志

**为什么选 stdio：**

1. Claude Code 原生支持 stdio 模式 MCP Server
2. 无需暴露网络端口
3. 进程生命周期由客户端管理

## 八、技术细节与难点

### Q: OCR 的准确率和性能如何？

**答：**

架构设计采用**双重后端 + 自动退化**（[src/scrape/ocr.py](src/scrape/ocr.py)）：

- 优先 `PaddleOCR(use_angle_cls=True, lang="ch")` — 中文场景准确率最优
- 回退 `easyocr.Reader(["ch_sim", "en"], gpu=False)` — 环境兼容性更好
- 两个都不可用时跳过 OCR，不阻塞主流程

**图片过滤优化：**

- `< 10KB` 或 `< 200x200px` 的图直接跳过（avatar、icon、缩略图）
- 文件大小 hash 去重，避免同一张图重复 OCR
- 只 OCR carousel 内的内容图片，排除 UI 装饰元素

**合并策略：** 多张图 OCR 后通过 `_has_overlap()` 检测相邻文本的重复片段（最少 15 字），有 overlap 则去重拼接，否则换行拼接。防止同一段文字被截成两张图后出现重复。

### Q: 整个流水线执行一次大概需要多长时间？

**答：** 以采集 5 篇帖子为例：

- Search：5-10s（页面加载 + 滚动加载 + 解析卡片）
- Scrape + OCR：15-30s（逐卡点击等待 6s + 图片下载 + OCR，每卡间隔 3-8s）
- Format（5 篇）：15-30s（DeepSeek API 串行调用，temperature=0.1）
- Classify（5 篇）：10-20s（DeepSeek API 串行调用，temperature=0）
- Build：5-10s（生成 MD 文件 + 向量化 + ChromaDB add）
- **总计**：约 50-100s

主要瓶颈在 Playwright 页面加载（每卡等 6s）和 DeepSeek API 串行调用。MCP `run_pipeline` Tool 设置了 600s 超时以应对大批量或网络波动。

### Q: ChromaDB 数据量增长后检索性能如何？

**答：**

- 当前设计每次 `build_index` 会 **delete + recreate** Collection，适合全量重建场景
- HNSW 索引查询复杂度约 O(log N)，千级数据量毫秒级响应
- 检索结果在 Python 层做了三层过滤：(1) 去重（by path）；(2) 排除 `INDEX.md` / `_index.md`；(3) 路径 scope 校验（只返回项目目录下的文件）
- 如果扩展到 10 万+ 级别，可考虑：不每次重建 Collection 而是增量 add、按 category 过滤缩小搜索空间、迁移到 Milvus 或 Qdrant

### Q: 遇到过哪些技术难点？怎么解决的？

**答：**

**难点 1：protobuf 版本兼容性**

- PaddleOCR 依赖的 `_pb2.py` 文件与 protobuf >= 4.x 不兼容，报 "Descriptors cannot be created directly"
- 解决：在**所有 import 之前**设置 `os.environ["PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION"] = "python"`，强制使用纯 Python protobuf 实现。这在 `xiaohongshu.py`、`mcp_server/xhs_server.py`、`src/scrape/ocr.py` 三处入口都有设置

**难点 2：笔记详情页结构多样化**

- 小红书帖子正文经过详情面板展示后，文本结构至少有 3 种变体（已关注/未关注/无关注按钮），字段顺序不同
- 解决：`_parse_note_text()` 实现了通用的逐行解析器，按 counter → author → 关注状态 → title → (subtitle) → body → #tags → date → comments 的弹性顺序解析，通过 `_find_tag_start()` 和 `_find_comment_start()` 动态定位分界点

**难点 3：LLM JSON 输出不稳定**

- Classify 步骤要求 LLM 返回 JSON，但 DeepSeek 有时会在 JSON 外包裹解释文字
- 解决：`_parse_classification_response()` 用 `raw.find("{")` / `raw.rfind("}")` 提取 JSON 片段后解析；解析失败则进 `_fallback_classify()` 纯规则分类（按内容长度和互动数估算质量分，从 tags 取前 5 个作关键词），保证流程不中断

**难点 4：OCR 结果与正文重复**

- LLM Format 阶段已把 OCR 文本合并到正文中，若 Markdown 生成时再单独输出 OCR section 会导致内容重复
- 解决：`_ocr_is_redundant()` 检查 OCR 文本各行是否已有 ≥50% 出现在格式化后的 content 中，重复则跳过 `## 图片提取文字` section（[src/knowledge\_base/builder.py](src/knowledge_base/builder.py#L102-L112)）

**难点 5：断点续抓**

- 批量抓取时网络波动或中途崩溃会导致重复劳动
- 解决：`output/checkpoint_scrape.json` 记录已完成的 post\_id，`scrape_posts()` 启动时加载并跳过已完成项，每完成一个立即 `_save_checkpoint()`

### Q: 你是怎么设计错误处理和容错的？

**答：**

### Q: MCP、CLI 和 Function Calling 的区别是什么？各自有什么优缺点？

**答：**

MCP（Model Context Protocol）是 Anthropic 提出的开放协议，基于 JSON-RPC 2.0，标准化 AI 应用与外部工具的交互。通信方式是通过 stdio 或 HTTP 传输，客户端启动 MCP Server 进程，Tool 调用通过协议传递。特点包括声明式 Tool 定义（JSON Schema）、协议标准化、一次开发多端使用。

CLI（命令行接口）通过 Shell 命令执行外部程序，获取 stdout 作为结果。通信方式是 subprocess 启动进程，命令和参数以字符串传入，结果以文本返回。特点是通用性强，任何有命令行的工具都能调用，但返回是纯文本需要自己解析。

Function Calling（函数调用）是 LLM 直接调用代码中定义的函数，本质是模型输出结构化的 JSON，程序执行对应函数返回结果。通信方式是模型输出函数名和参数，程序执行后返回结果给模型。特点是模型原生能力，低延迟，但需要为每个任务预先定义函数签名。

三个方案的对比表格如下：

| 维度      | MCP         | CLI        | Function Calling |
| ------- | ----------- | ---------- | ---------------- |
| 标准化     | 是（开放协议）     | 否          | 否（各模型不同）         |
| 通信方式    | stdio/HTTP  | subprocess | 模型输出 JSON        |
| Tool 定义 | JSON Schema | 无          | 代码函数签名           |
| 适用场景    | 通用工具调用      | 系统命令、脚本    | 模型决策触发动作         |
| 延迟      | 中等（进程通信）    | 高（进程启动开销）  | 低（直接函数调用）        |
| 扩展性     | 好（协议驱动）     | 好（任意命令）    | 一般（需改代码）         |
| 开发成本    | 中等          | 低          | 低                |
| 多模态支持   | 支持          | 不支持        | 不支持              |

MCP 的优点是协议标准化、可复用、支持多模态生态好，缺点是概念抽象、需要理解协议层。CLI 的优点是通用性强、任何命令行工具都能用、实现简单，缺点是返回纯文本需要解析、不支持流式交互。Function Calling 的优点是模型原生、低延迟、实现简单，缺点是跨任务能力有限、需要预先定义函数、模型能力决定效果。

在项目中的体现：MCP Server 封装为本项目的 MCP 工具，供 Claude Code 等 Agent 调用；CLI 支持命令行执行搜索、采集，建库等操作；LLM（DeepSeek）在 Format 和 Classify 阶段输出结构化 JSON 作为函数调用结果。

### Q: 介绍一下从用户提问到返回答案的链路？

**答：**

用户提问到返回答案的完整链路分为两个阶段。

第一阶段是检索（Retrieval）。首先对用户 query 进行 jieba 分词，得到关键词列表；然后并行执行两路召回，关键词召回匹配 frontmatter 的 tags 和 keywords 字段，语义召回通过 GTE Embedding 向量化后由 ChromaDB 做 cosine similarity 搜索；接着两路结果合并去重，同时命中的取两路分数平均并标记为 hybrid；最后按分数降序取 top_k 返回。

第二阶段是格式化返回。调用 format_results() 函数，将结果格式化为 Markdown 展示，包含标题、分数、路径、摘要、来源和原文链接。

本项目是一个 RAG 检索系统，不是生成式问答系统。链路到返回检索结果就结束了，没有 LLM 生成答案这一步。search_kb 工具返回的是相关文档的标题、路径、摘要、相似度分数和原文链接。如果需要生成答案，需要在检索后额外加一层，将检索内容拼装成 Prompt 发给 LLM 生成答案，这个生成步骤在本项目中没有实现，是后续可以扩展的方向。

本项目链路与完整 RAG 链路的对比如下：

| 阶段 | 本项目       | 完整 RAG（带生成）  |
| -- | --------- | ------------ |
| 检索 | 有         | 有            |
| 增强 | 无（直接返回结果） | 有（拼装 Prompt） |
| 生成 | 无         | 有（LLM 生成答案）  |

### Q: 介绍一下 RAG 部分的框架？

**答：**

RAG 部分的框架分为两个核心层次。框架结构图如下：

```
┌─────────────────────────────────────────────────────┐
│                    用户 Query                          │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│                 检索层（RAG Engine）                   │
│  ┌─────────────┐          ┌─────────────────────┐   │
│  │ 关键词召回   │          │ 语义召回 (ChromaDB)   │   │
│  │ jieba 分词  │          │ GTE-small Embedding  │   │
│  │ 匹配 tags   │          │ HNSW 向量索引          │   │
│  └──────┬──────┘          └──────────┬──────────┘   │
│         │                              │              │
│         └──────────┬─────────────────┘              │
│                    ▼                                 │
│            ┌──────────────┐                         │
│            │  分数融合合并  │  avg(hybrid) / keep     │
│            └──────┬───────┘                         │
│                   ▼                                  │
│            返回 Top-K 检索结果                         │
└─────────────────────┬───────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────┐
│                  结果格式化层                         │
│         format_results() → Markdown                   │
└─────────────────────────────────────────────────────┘
```

检索层（RAG Engine）包含两路并行召回。关键词召回使用 jieba 分词对 query 进行切词，然后匹配 frontmatter 的 tags 和 keywords 字段得到关键词得分。语义召回使用 GTE-small Embedding 将 query 向量化后，通过 ChromaDB 的 HNSW 向量索引做 cosine similarity 搜索得到语义得分。两路结果合并去重，同时命中的取两路分数平均（hybrid），只有一路命中的保留各自分数，最终按分数降序取 top_k。

结果格式化层调用 format_results() 函数，将检索结果格式化为 Markdown 展示，包含标题、分数、路径、摘要、来源和原文链接。

核心模块及职责如下：

| 模块   | 文件                       | 职责                                                   |
| ---- | ------------------------ | ---------------------------------------------------- |
| 索引构建 | kb_agent/indexer.py     | 解析 MD、按 ## 切 Chunk、GTE 向量化、入库 ChromaDB               |
| 检索引擎 | kb_agent/rag_engine.py | keyword_search、semantic_search、hybrid_search 三种检索 |
| 检索协调 | kb_agent/searcher.py    | 统一入口、结果格式化、scope 校验                                  |

建库流程为：MD 文件经过 _parse_md() 解析，按 ## 二级标题切 Chunk，通过 _get_embedding() 生成向量，最后调用 collection.add() 入库。检索流程为：query 触发 keyword_search() 和 semantic_search() 并行执行，结果经 hybrid_merge() 合并，最后由 format_results() 格式化输出。

技术选型及原因如下：

| 组件        | 选型                   | 原因                 |
| --------- | -------------------- | ------------------ |
| 向量数据库     | ChromaDB             | 轻量、嵌入式、本地持久化       |
| Embedding | ModelScope GTE-small | 中文优化、本地运行、无 API 成本 |
| 关键词检索     | jieba 分词             | 轻量、精准匹配、互补语义召回     |
| 融合策略      | 简单平均                 | 分数量纲相同，无需 RRF      |

### Q: Chunk 切片太小或者太大会有什么问题？

**答：**

Chunk 太小会产生三方面的问题。语义不完整方面，每个 Chunk 成为独立检索单元，如果切得太碎可能丢失段落内上下文依赖，LLM 只能看到孤立的一句话无法理解完整语义。召回噪音增加方面，检索会返回大量细碎 Chunk，top_k 条结果可能来自十几篇不同文章，用户需要在大海里捞针，相关内容反而被稀释。上下文窗口浪费方面，LLM 收到多个不相关的短片段，大部分 token 花在拼凑上下文上，生成质量下降。

Chunk 太大会产生三方面的问题。语义偏移方面，一个很长 Chunk 可能包含多个主题，比如一篇 3000 字面经涵盖算法题、系统设计、HR 问题三个部分，用户只问了系统设计但整篇 Chunk 被召回，前面算法题内容对用户是噪音。Embedding 精度下降方面，大多数 Embedding 模型有 512 token 输入限制，超长文本会被截断，embedding 无法准确表示全文语义。相关内容被稀释方面，多主题 Chunk 中真正相关的段落淹没在大量无关内容里。

太小和太大的问题总结对比如下：

| 问题 | 太小 | 太大 |
|------|------|------|
| 语义完整性 | 差（上下文丢失） | 好 |
| 检索精度 | 好（精准匹配） | 差（噪音多） |
| Embedding 质量 | 好（主题集中） | 差（可能被截断） |
| 召回噪音 | 多（结果碎片化） | 多（多主题混杂） |

最佳实践中，固定长度分块建议 200-500 字每块，相邻块重叠 50-100 字避免语义切断；按语义切分按段落或标题层级切分效果更好但成本高。
